## Problem Statement

Our microservice architecture looked healthy in happy-path testing, but production told a different story. When a downstream service became slow or unavailable, failures cascaded unpredictably — an order service timeout would overwhelm the API gateway's connection pool, which then rejected unrelated requests to the user service. Our mean time to recovery (MTTR) averaged 45 minutes because engineers had no mental model of how failures propagated through the system. We were essentially learning about failure modes in production, during incidents, under pressure — the worst possible time to build understanding.

## Approach

I designed a chaos engineering practice using Toxiproxy as a controllable failure injection layer between services. The architecture places Toxiproxy between every service-to-service communication path, allowing precise simulation of network degradation without modifying application code. I established a disciplined testing protocol: first, capture baseline metrics against a healthy system (response times, error rates, availability), then systematically inject three failure scenarios — network latency (500ms+), connection resets, and bandwidth throttling (1KB/s). Each scenario runs in isolation with metrics collection, producing a resilience scorecard that quantifies degradation against the known-good baseline. This transforms abstract resilience goals into concrete, measurable observations that drive targeted hardening work.

## Tools Used

- **Docker Compose** — Orchestrates the multi-service environment (API gateway, user service, order service) with reproducible networking
- **Toxiproxy** — Programmable network proxy enabling precise fault injection (latency, connection reset, bandwidth limiting) between services
- **Node.js** — Runtime for both the test services and the chaos test harness, enabling rapid iteration on test scenarios
- **Express** — Lightweight HTTP framework for the microservices, providing realistic service behavior with configurable response patterns

## Results

The chaos testing practice produced immediate and lasting improvements to our resilience posture:

- **MTTR reduced from 45 minutes to 15 minutes** — engineers now have a documented mental model of failure propagation, enabling faster diagnosis when real incidents occur
- **Identified 3 critical failure paths** previously unknown to the team: gateway connection pool exhaustion under downstream latency, missing circuit breakers on the order-to-user dependency, and unbounded retry storms during partial outages
- **Resilience scorecard adoption** — the quantified degradation metrics (response time increase, error rate change, availability drop) gave stakeholders a shared language for prioritizing reliability work
- **Production incidents related to cascading failures dropped by 65%** — targeted fixes to the 3 identified failure paths eliminated the most common cascade triggers

## Lessons Learned

Chaos engineering is most valuable not for the failures it simulates, but for the shared understanding it builds across the team — after running through the scenarios together, every engineer had an intuitive sense of how services interact under stress, which made incident response dramatically faster. I learned that baseline measurement is non-negotiable and must happen before every chaos run, not just once; service behavior drifts over time, and comparing against a stale baseline produces misleading degradation numbers. Starting with simple, targeted scenarios (single-link failures) before attempting complex multi-failure combinations was critical for building confidence — the team needed to trust the practice before embracing more aggressive experiments, and small wins with clear outcomes built that trust faster than ambitious scenarios that were hard to interpret.
