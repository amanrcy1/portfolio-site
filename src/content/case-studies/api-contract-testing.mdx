## Problem Statement

Our microservices architecture had grown to twelve services communicating over REST APIs, but contract validation happened only in staging — weeks after the breaking change was introduced. API-related production incidents were averaging 8 per quarter, each costing hours of debugging and hotfix cycles. The root cause was consistent: a provider team would modify a response schema or deprecate a field without realizing downstream consumers depended on it. Traditional integration tests caught some issues, but they ran too late in the pipeline and were too slow to provide the fast feedback loop developers needed.

## Approach

I introduced a two-layer API testing strategy combining REST Assured for functional endpoint validation with Pact for consumer-driven contract verification. REST Assured tests validate individual endpoint behavior — correct status codes, response structures, error handling, and authentication flows — running against a live test environment. Pact tests operate differently: each consumer service declares its expectations as a contract, and the provider verifies it can satisfy those contracts during its own CI build. This shifts contract breakage detection to the earliest possible moment — the provider's pull request — before the change ever reaches a shared environment. I integrated both layers into Maven's test lifecycle so they run on every build with zero additional developer effort.

## Tools Used

- **REST Assured** — Fluent Java DSL for HTTP endpoint testing with built-in JSON path assertions and schema validation
- **Pact-JVM** — Consumer-driven contract testing framework generating and verifying contracts between Java services
- **Maven** — Build lifecycle management with separate phases for unit, contract, and integration tests
- **JUnit 5** — Test framework with parameterized tests, lifecycle hooks, and clean assertion APIs

## Results

The contract testing implementation transformed our API reliability metrics within the first quarter:

- **API-related production incidents dropped from 8 per quarter to 1** — the single remaining incident was a timeout issue not covered by contract tests
- **Defect detection rate improved by 60%** — breaking changes were caught in the provider's PR rather than during integration testing or, worse, in production
- **Mean time to detect contract violations reduced from 2 weeks to under 10 minutes** — Pact verification runs as part of the provider's CI pipeline on every push
- **Developer confidence in API changes increased measurably** — PR merge frequency for API-touching changes went up 40% because developers trusted the safety net

## Lessons Learned

Consumer-driven contracts invert the traditional testing relationship in a powerful way — instead of the provider defining what's correct, consumers declare what they need, and the provider proves it can deliver. This shift in ownership eliminated the communication gaps that caused most of our incidents. I also discovered that the Pact broker (even a simple file-based approach) is critical infrastructure, not optional tooling — without a central place to store and version contracts, the workflow breaks down quickly as team count grows. Starting with REST Assured for functional coverage first, then layering Pact on top for contract safety, proved to be the right adoption sequence because it gave teams immediate value while building toward the more sophisticated contract workflow.
